## Архитектура

```
┌─────────────────────────────────────────────────────────────────┐
│                     ЦЕНТРАЛЬНЫЙ СЕРВЕР                           │
│                                                                  │
│  Кадровик/Суперадмин                                            │
│       │ нажимает "Открыть доступ"                               │
│       ▼                                                          │
│  DoorCommand { action: "grant", status: "pending" }              │
│       │                                                          │
│  Telegram: "🟢 Доступ ВЫДАН (ожидает агента)"                   │
└───────────────────────┬─────────────────────────────────────────┘
                        │
                   HTTPS (исходящий)
                   Агент сам инициирует
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │  Офис 1  │    │  Офис 2  │    │  Офис 3  │
  │          │    │          │    │          │
  │  Agent   │    │  Agent   │    │  Agent   │
  │ cId = 1  │    │ cId = 2  │    │ cId = 3  │
  └────┬─────┘    └────┬─────┘    └──────────┘
       │ HTTP локал.   │
  ┌────┴───────┐       │
  │ 192.168.x  │  [аналогично]
  │            │
  │  IN device │ ← Face ID снаружи (вход)
  │ OUT device │ ← Face ID внутри  (выход)
  └────────────┘
```

**Ключевые принципы:**
- Устройства **никогда не торчат в интернет**
- Агент использует только **исходящий** HTTPS к серверу
- Работает через любой NAT, мобильный интернет, без VPN


### 2. Таблицы БД (создаются автоматически через Prisma)

Если нужно создать вручную:
```sql
-- Двери
CREATE TABLE Door (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  companyId INT NOT NULL,
  inDeviceIp VARCHAR(255) NOT NULL,
  inDevicePort INT DEFAULT 80,
  outDeviceIp VARCHAR(255) NOT NULL,
  outDevicePort INT DEFAULT 80,
  login VARCHAR(255) DEFAULT 'admin',
  password VARCHAR(255) NOT NULL,
  isActive BOOLEAN DEFAULT TRUE,
  createdAt DATETIME(3) DEFAULT CURRENT_TIMESTAMP(3),
  updatedAt DATETIME(3) DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  INDEX idx_companyId (companyId),
  FOREIGN KEY (companyId) REFERENCES Company(id)
);

-- Доступы сотрудников к дверям
CREATE TABLE DoorAccess (
  id INT AUTO_INCREMENT PRIMARY KEY,
  doorId INT NOT NULL,
  employeeId INT NOT NULL,
  grantedBy VARCHAR(255) NOT NULL,
  createdAt DATETIME(3) DEFAULT CURRENT_TIMESTAMP(3),
  UNIQUE KEY uniq_door_employee (doorId, employeeId),
  FOREIGN KEY (doorId) REFERENCES Door(id) ON DELETE CASCADE,
  FOREIGN KEY (employeeId) REFERENCES Employee(id) ON DELETE CASCADE
);

-- Очередь команд для relay-агента
CREATE TABLE DoorCommand (
  id INT AUTO_INCREMENT PRIMARY KEY,
  doorId INT NOT NULL,
  employeeId INT NOT NULL,
  action VARCHAR(10) NOT NULL,       -- 'grant' | 'revoke'
  status VARCHAR(20) DEFAULT 'pending', -- pending | processing | done | failed
  error TEXT,
  createdAt DATETIME(3) DEFAULT CURRENT_TIMESTAMP(3),
  updatedAt DATETIME(3) DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
  INDEX idx_status_doorId (status, doorId),
  FOREIGN KEY (doorId) REFERENCES Door(id) ON DELETE CASCADE,
  FOREIGN KEY (employeeId) REFERENCES Employee(id) ON DELETE CASCADE
);
```

## Управление дверями

Доступно только **Суперадмину** через `/doors`.

### Что такое "дверь" в системе

Одна дверь = **два физических Hikvision устройства**:
- **IN device** — Face ID снаружи (сотрудник прикладывает лицо на вход)
- **OUT device** — Face ID внутри (сотрудник прикладывает лицо на выход)

### Создание двери

1. Войти как Суперадмин → меню "Двери (СКУД)"
2. Нажать "Добавить дверь"
3. Заполнить:
   - **Название** — например "Главный вход", "Склад", "Серверная"
   - **Компания** — к какой компании относится дверь
   - **IP и порт IN устройства** — IP адрес Face ID снаружи (192.168.x.x)
   - **IP и порт OUT устройства** — IP адрес Face ID внутри
   - **Логин и пароль** — учётные данные Hikvision (по умолчанию admin/12345)

### Получить IP устройства Hikvision

1. Hikvision SADP Tool → найдёт устройства в сети автоматически
2. Или: роутер → список DHCP клиентов → найти по MAC адресу

## Выдача доступа сотрудникам

Доступно **Кадровику** и **Суперадмину**.

### Через профиль сотрудника

1. Открыть профиль сотрудника (`/employees/[id]`)
2. Перейти на вкладку **"Доступ к дверям"**
3. Нажать **"Открыть доступ"** напротив нужной двери

Система немедленно:
- Создаёт запись в `DoorAccess`
- Ставит команду `grant` в очередь `DoorCommand`
- Отправляет Telegram уведомление

Relay-агент в офисе выполняет команду в течение ~5 секунд:
- Добавляет пользователя в Hikvision (UserInfo)
- Загружает фото лица (FaceDataRecord)
- Применяет на оба устройства (IN и OUT)
- Отправляет подтверждение на сервер

### Отзыв доступа

Нажать **"Закрыть доступ"** — агент удалит пользователя с обоих устройств.

### Что нужно для работы Face ID

Сотрудник должен иметь **фото в системе** (загружается через профиль → кнопка загрузки фото). Без фото пользователь будет добавлен на устройство, но идентификация по лицу работать не будет.

## Relay-агент в офисе

Подробная инструкция по установке: **[../agent/README.md](../agent/README.md)**

### Быстрая установка

1. Скопировать папку `agent/` на компьютер в офисе
2. Запустить **`УСТАНОВИТЬ.bat`** от имени администратора
3. Ввести 3 параметра: адрес сервера, токен, ID компании
4. Готово

### Как работает агент

```
Каждые 5 секунд:
  GET /api/agent/commands?companyId=X
  → Если есть команды:
      Для каждой команды:
        - Скачать фото сотрудника с сервера
        - Выполнить ISAPI на IN device
        - Выполнить ISAPI на OUT device
        - PATCH /api/agent/commands/:id { status: "done" }

Каждые 30 секунд (мониторинг):
  Для каждого устройства компании:
    TCP connect к IP:порт (таймаут 4 сек)
    Если состояние изменилось → POST /api/agent/device-alert
```

### Статусы команд

| Статус | Значение |
|--------|---------|
| `pending` | Создана, ожидает агента |
| `processing` | Агент забрал, выполняет |
| `done` | Выполнена успешно |
| `failed` | Ошибка (текст в поле `error`) |

### Сброс зависших команд

Если агент упал во время выполнения, команды останутся в статусе `processing`. Сбросить:

```sql
UPDATE DoorCommand
SET status = 'pending'
WHERE status = 'processing'
  AND updatedAt < DATE_SUB(NOW(), INTERVAL 10 MINUTE);
```

## Мониторинг устройств

Агент проверяет TCP-доступность каждого устройства каждые **30 секунд** (настраивается через `deviceCheckIntervalMs` в `config.json`).

**Логика алертов:**
- При **первом запуске** — записывает начальное состояние, без уведомлений
- При **изменении состояния** (онлайн→офлайн или офлайн→онлайн) — отправляет алерт
- Если устройство остаётся в том же состоянии — уведомлений нет (нет спама)

**Проверка** выполняется TCP connect (не ping, не HTTP) — это надёжнее всего.

## Telegram уведомления

| Событие | Иконка | Описание |
|---------|--------|---------|
| Новая дверь добавлена | 🚪 | Название, IP, компания, кто добавил |
| Дверь изменена (вкл/выкл) | ⚙️ | Новый статус, кто изменил |
| Дверь удалена | 🗑 | Название, кто удалил |
| Доступ выдан | 🟢 | Сотрудник, дверь, кто выдал |

| Агент выполнил (grant) | ✅ | Подтверждение добавления на устройство |
| Агент выполнил (revoke) | ✅ | Подтверждение удаления с устройства |
| Ошибка агента | ❌ | Текст ошибки, рекомендация проверить устройство |
| Устройство офлайн | 🔴 | Дверь, IP, компания |
| Устройство восстановлено | 🟢 | Дверь, IP, компания |

## API эндпоинты

### Для фронтенда (JWT авторизация)

| Метод | URL | Роль | Описание |
|-------|-----|------|---------|
| GET | `/api/doors?companyId=X` | Суперадмин, Кадровик, Руководитель | Список дверей |
| POST | `/api/doors` | Суперадмин | Создать дверь |
| PATCH | `/api/doors/:id` | Суперадмин | Обновить дверь |
| DELETE | `/api/doors/:id` | Суперадмин | Удалить дверь |
| GET | `/api/doors/employee/:id` | Суперадмин, Кадровик, Руководитель | Двери сотрудника с флагом hasAccess |
| POST | `/api/doors/:id/grant/:empId` | Суперадмин, Кадровик | Выдать доступ |
| DELETE | `/api/doors/:id/revoke/:empId` | Суперадмин, Кадровик | Отозвать доступ |

### Для relay-агента (заголовок `X-Agent-Token`)

| Метод | URL | Описание |
|-------|-----|---------|
| GET | `/api/agent/commands?companyId=X` | Получить pending команды |
| PATCH | `/api/agent/commands/:id` | Обновить статус команды |
| GET | `/api/agent/doors?companyId=X` | Список дверей для мониторинга |
| GET | `/api/agent/photo/:employeeId` | Скачать фото сотрудника |
| POST | `/api/agent/device-alert` | Сообщить об изменении состояния устройства |
| GET | `/api/agent/status?companyId=X` | Статистика команд |

## Устранение неисправностей

### Команда висит в статусе `pending`

Агент не запущен или не может достучаться до сервера.

```bash
# В офисе — проверить что агент запущен
tasklist | findstr node

# Проверить логи агента
type C:\Hikvision\agent.log

# Запустить вручную для теста
cd C:\Hikvision
node relay-agent.js
```

### Команда в статусе `failed`

Агент не смог подключиться к Hikvision устройству. Telegram пришлёт ошибку.

Возможные причины:
- Неверный IP устройства — проверить в настройках двери
- Устройство выключено или перезагружается
- Неверный пароль Hikvision — проверить в настройках двери
- Агент и устройство в разных сетях

### Лицо не распознаётся

1. Убедиться что у сотрудника есть фото в системе (профиль → фото)
2. Проверить в логах агента что фото было загружено (`Фото загружено X KB`)
3. Фото должно быть чётким, лицо хорошо видно

### Telegram не приходит уведомление

Проверить в `backend/.env`:
```env
TELEGRAM_TOKEN=xxxxxxx:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
TELEGRAM_CHAT_IDS=123456789,987654321
```

Проверить логи бэкенда:
```bash
docker compose logs backend | grep -i telegram
```

### Устройство показывает офлайн хотя работает

Проверить что агент находится **в той же локальной сети** что и устройства. TCP connect выполняется с машины агента.
