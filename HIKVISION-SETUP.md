## 1. Как устроена система

```
┌──────────────────────────────────────────────────────────────┐
│                 ЦЕНТРАЛЬНЫЙ СЕРВЕР (публичный IP)             │
│                  NestJS Backend + MySQL                       │
│                                                              │
│   ┌──────────────────┐    ┌────────────────────────────┐    │
│   │  /api/hikvision   │    │  /api/agent/commands       │    │
│   │  /event (webhook) │    │  (команды для relay-agent) │    │
│   └────────┬─────────┘    └─────────────┬──────────────┘    │
└────────────┼──────────────────────────────┼──────────────────┘
             │                              │
             │ HTTP POST                    │ HTTPS polling
             │ (событие Face ID)            │ каждые 5 сек
             │                              │
    ┌────────▼──────────────────────────────▼────────────────┐
    │                  ОФИС (локальная сеть)                  │
    │                                                         │
    │  ┌─────────────────┐      ┌───────────────────────┐    │
    │  │ Hikvision Face  │      │   relay-agent.js      │    │
    │  │ устройство ВХОД │◄─────│  (мини-ПК / RPi)      │    │
    │  │ 192.168.1.101   │      │                       │    │
    │  └─────────────────┘      └───────────────────────┘    │
    │  ┌─────────────────┐             │                      │
    │  │ Hikvision Face  │             │ ISAPI HTTP           │
    │  │ устройство ВЫХОД│◄────────────┘                      │
    │  │ 192.168.1.102   │                                    │
    │  └────────┬────────┘                                    │
    └───────────┼─────────────────────────────────────────────┘
                │ webhook (HTTP POST к серверу)
                └──────────────────────────────────────────────▶ Сервер
```

**Ключевые принципы:**
- Hikvision устройства **никогда не торчат в интернет** — они только в локальной сети
- Relay-agent использует **только исходящие** соединения (опрашивает сервер сам)
- Сервер никогда не подключается к устройствам напрямую — только через агента
- Роутер офиса не требует никакого проброса портов для агента

## 2. Два канала связи

Система работает через **два независимых канала**. Важно понять: они независимы.

### Канал A — Посещаемость и Telegram (Hikvision → Сервер)

| Что делает | Устройство срабатывает на лицо → шлёт событие на сервер |
|-----------|-------------------------------------------------------|
| Что получаем | Запись в БД (время, дата, вход/выход), Telegram уведомление |
| Нужен ли relay-agent | **НЕТ** |
| Нужен ли компьютер в офисе | **НЕТ** |
| Требование к сети | У устройства должен быть выход в интернет к серверу |

### Канал B — Управление Face ID (Сервер → relay-agent → Hikvision)

| Что делает | Выдача/отзыв доступа, загрузка фото на устройство |
|-----------|-------------------------------------------------------|
| Что получаем | Сотрудник добавлен/удалён в Face ID устройства |
| Нужен ли relay-agent | **ДА** |
| Нужен ли компьютер в офисе | ДА (но достаточно мини-ПК за $30-50, см. ниже) |
| Требование к сети | Агент должен видеть сервер (интернет) и устройства (локальная сеть) |

## 3. Канал 1 — Посещаемость и Telegram

### Что происходит когда сотрудник прикладывает лицо:

```
Hikvision распознаёт лицо
    ↓
HTTP POST → https://сервер/api/hikvision/event
    ↓
Сервер определяет офис по IP устройства (HIKVISION_DEVICES в .env)
    ↓
Ищет сотрудника по skudId (Employee.skudId = employeeNo из устройства)
    ↓
Создаёт AttendanceEvent (timestamp, direction=IN/OUT, officeId, source=HIKVISION)
    ↓
Пересчитывает дневную сводку (время прихода, ухода, опоздание)
    ↓
Telegram: "🟢 Вход сотрудника | Офис: Главный | Иванов Иван | 09:05"
```

### Telegram уведомление — формат:

```
🟢 Вход сотрудника

🏢 Офис: Главный офис
🚪 Дверь: Вход (Снаружи)
👤 Сотрудник: Иванов Иван Петрович
⏰ Время: 17.03.2025, 09:05:32
```

При выходе:
```
🔴 Выход сотрудника

🏢 Офис: Главный офис
🚪 Дверь: Выход (Внутри)
👤 Сотрудник: Иванов Иван Петрович
⏰ Время: 17.03.2025, 18:30:11
```

Если лицо не найдено в системе (неизвестный):
```
⚠️ Неизвестный сотрудник
СКУД №: 00042
Устройство: Главный офис (Вход)
IP: 192.168.1.101
```

### Для этого канала нужно только:

1. Прописать устройства в `HIKVISION_DEVICES` на сервере (см. раздел 7)
2. На самом устройстве Hikvision настроить HTTP Notification на адрес сервера (см. раздел 9)
3. **Всё.** Компьютер в офисе не нужен.

## 4. Канал 2 — Управление Face ID (relay-agent)

### Что происходит когда кадровик выдаёт доступ:

```
Кадровик → нажимает "Выдать доступ" на странице /doors
    ↓
Сервер: создаёт DoorCommand { action: "grant", status: "pending" }
    ↓
Relay-agent в офисе (раз в 5 секунд): "есть команды?" → GET /api/agent/commands
    ↓
Получает команду → скачивает фото сотрудника с сервера
    ↓
Hikvision ISAPI: PUT /ISAPI/AccessControl/UserInfo/Record (добавить пользователя)
Hikvision ISAPI: PUT /ISAPI/Intelligent/FDLib/FaceDataRecord (загрузить фото лица)
    ↓
Агент отчитывается серверу: PATCH /api/agent/commands/:id { status: "done" }
    ↓
Telegram: "✅ Доступ выдан: Иванов → Дверь Главного офиса"
```

### Без компьютера — возможные варианты:

| Вариант | Стоимость | Сложность | Описание |
|---------|-----------|-----------|---------|
| **Raspberry Pi Zero 2W** | ~$20 + блок питания | Низкая | Маленький Linux-компьютер, Node.js, работает без монитора и клавиатуры, питается от USB |
| **Raspberry Pi 4B** | ~$50 | Низкая | Мощнее, более надёжен для постоянной работы |
| **Мини-ПК N100** | ~$80-120 | Нулевая | Обычный Windows ПК, просто маленький. Полный контроль, легко обслуживать |
| **Роутер OpenWrt** | 0 (уже есть) | Высокая | Если роутер поддерживает opkg + Node.js (не все) |
| **Старый ноутбук/ПК** | 0 | Нулевая | Закрыть крышку, поставить в угол |

**Рекомендация**: Raspberry Pi Zero 2W или мини-ПК N100 — ставится один раз, не требует обслуживания, потребляет 5-15 Вт.

**Важно**: Если канал A (посещаемость) для вас важнее — можно запустить систему сначала только с webhook и добавить агентов позже, когда нужна выдача/отзыв Face ID.

## 5. Что нужно на каждый офис

### Только посещаемость (без управления Face ID):

```
✅ Hikvision устройства с прошивкой V5.x+ (поддержка HTTP Notification)
✅ У устройств есть выход в интернет (нужен только порт 80/443 наружу)
✅ Статический локальный IP у каждого устройства (назначь в роутере по MAC)
✅ Прописать устройства в HIKVISION_DEVICES на сервере
✅ На устройстве настроить Event → HTTP Notification → адрес сервера
```

### Полная интеграция (посещаемость + управление Face ID):

```
Всё из выше, плюс:
✅ Один маленький компьютер/Raspberry Pi в каждом офисе (или в одном офисе на компанию)
✅ На нём установлен Node.js 18+ и relay-agent
✅ config.json заполнен: serverUrl, agentToken, companyId
✅ Запущен как автозапуск (install.bat)
```

## 6. Как добавить дверь в систему

### Через интерфейс (суперадмин only):

1. Войти как суперадмин
2. Перейти в меню **СКУД / Двери** (`/doors`)
3. Нажать **"Добавить дверь"**
4. Заполнить форму:

| Поле | Значение | Пример |
|------|----------|--------|
| Название | Человекочитаемое | `Главный вход` |
| Компания | Выбрать из списка | `Бунёд Интернешнл` |
| IP устройства ВХОДА | Локальный IP Hikvision снаружи | `192.168.1.101` |
| Порт ВХОДА | Обычно 80 | `80` |
| IP устройства ВЫХОДА | Локальный IP Hikvision внутри | `192.168.1.102` |
| Порт ВЫХОДА | Обычно 80 | `80` |
| Логин | Логин на устройстве | `admin` |
| Пароль | Пароль на устройстве | `Admin12345` |

5. Нажать **"Сохранить"**

### Как узнать IP устройства Hikvision:

**Способ 1 — SADP утилита Hikvision (рекомендуется)**
- Скачать бесплатно: [Hikvision SADP Tool](https://www.hikvision.com/en/support/tools/hitools/sadp/)
- Запустить на любом компьютере в той же сети
- Покажет все Hikvision устройства с IP, MAC, серийным номером

**Способ 2 — Роутер**
- Зайти в панель роутера → DHCP клиенты
- Найти устройства с именем `HIK...` или производитель Hikvision

**Способ 3 — Напрямую на устройстве**
- На устройстве Hikvision есть меню (если есть экран): System → Network → IP Address

### Закрепить статический IP (делается один раз):

В роутере найди раздел **DHCP → Static Lease / Address Reservation**:

```
MAC-адрес устройства → Зарезервированный IP

AA:BB:CC:DD:EE:01  →  192.168.1.101  (Вход)
AA:BB:CC:DD:EE:02  →  192.168.1.102  (Выход)
```

MAC-адрес видно в SADP утилите или на наклейке на устройстве.
После этого IP никогда не изменится, даже если устройство перезагрузится.

## 7. Настройка HIKVISION_DEVICES на сервере

Это **обязательная** настройка для работы посещаемости. Без неё сервер не будет знать что делать с событиями от устройств.

### Открой файл `backend/.env` на сервере:

```env
HIKVISION_DEVICES=[
  {"ip":"192.168.1.101","officeName":"Главный офис","direction":"IN"},
  {"ip":"192.168.1.102","officeName":"Главный офис","direction":"OUT"},
  {"ip":"192.168.2.101","officeName":"Склад","direction":"IN"},
  {"ip":"192.168.2.102","officeName":"Склад","direction":"OUT"}
]
```

### Правила:

| Поле | Описание |
|------|---------|
| `ip` | Локальный IP устройства Hikvision (тот же что ты вписал при добавлении двери) |
| `officeName` | Название офиса — **должно точно совпадать** с названием в системе (Настройки → Офисы) |
| `direction` | `"IN"` — устройство снаружи (считывает вход), `"OUT"` — изнутри (выход) |

### Что происходит если IP не в списке:

Сервер получит событие, увидит неизвестный IP и:
- Напишет в лог: `Неизвестный IP устройства: 192.168.x.x — добавьте в HIKVISION_DEVICES`
- Отправит Telegram: `⚠️ Неизвестный сотрудник` (если лицо было распознано)
- Событие посещаемости **не запишется**

### После изменения .env — перезапусти backend:

```bash
docker-compose up -d backend
```

## 8. Установка relay-agent в офисе

> Нужно только если хочешь управлять Face ID (выдача/отзыв доступа).
> Для простой записи посещаемости — этот раздел пропускай.

### Подготовка компьютера/Raspberry Pi:

1. Установить **Node.js 18+** (nodejs.org)
2. Скопировать папку `agent/` из репозитория на компьютер, например в `C:\hrms-agent\`

### Создать config.json:

Скопировать `config.example.json` → `config.json` и заполнить:

```json
{
  "serverUrl": "https://твой-домен.com",
  "agentToken": "значение_AGENT_SECRET_TOKEN_из_.env_сервера",
  "companyId": 3,
  "pollIntervalMs": 5000,
  "deviceCheckIntervalMs": 30000
}
```

**Как узнать agentToken**: на сервере открой `backend/.env`, найди строку `AGENT_SECRET_TOKEN=...`

**Как узнать companyId**: войди в систему как суперадмин → `/admin` → найди нужную компанию

### Проверка перед установкой:

```bat
node test-connection.js
```

Должно показать:
```
✅ Сервер: доступен (команд в очереди: 0)
✅ Устройство IN  192.168.1.101:80 — онлайн
✅ Устройство OUT 192.168.1.102:80 — онлайн
```

### Установка как автозапуск (Windows):

```
Правый клик на install.bat → Запуск от имени администратора
```

Агент будет:
- Запускаться автоматически с Windows (даже без входа в систему)
- Перезапускаться автоматически при сбое
- Работать под системной учётной записью
- Логировать всё в `agent.log`

### Установка на Raspberry Pi (Linux):

```bash
# Установить Node.js 18
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt-get install -y nodejs

# Скопировать агента
mkdir ~/hrms-agent && cd ~/hrms-agent
# (скопировать файлы agent/ сюда)

# Создать config.json (заполнить значения)
cp config.example.json config.json
nano config.json

# Установить как systemd сервис (автозапуск)
sudo nano /etc/systemd/system/hrms-agent.service
```

Содержимое сервиса:
```ini
[Unit]
Description=HRMS Relay Agent
After=network.target

[Service]
Type=simple
User=pi
WorkingDirectory=/home/pi/hrms-agent
ExecStart=/usr/bin/node relay-agent.js
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable hrms-agent
sudo systemctl start hrms-agent
sudo systemctl status hrms-agent
```

### Управление агентом:

```bat
:: Windows — запустить вручную (для теста)
node relay-agent.js

:: Windows — посмотреть статус
schtasks /query /tn "HRMS Relay Agent"

:: Windows — просмотр логов (последние 50 строк)
powershell Get-Content agent.log -Tail 50

:: Linux — статус
sudo systemctl status hrms-agent

:: Linux — логи в реальном времени
journalctl -u hrms-agent -f
```

## 9. Настройка самих устройств Hikvision

Это делается один раз на каждом устройстве через браузер.

### Шаг 1 — Зайти на устройство

В браузере открыть: `http://192.168.1.101` (IP устройства)
Логин: `admin`, пароль: тот что стоит на устройстве (или `Admin12345` если не меняли)

### Шаг 2 — Настроить HTTP Notification (для посещаемости)

**Путь**: Configuration → Network → Advanced Settings → HTTP Listening

| Поле | Значение |
|------|---------|
| Enable | ✅ Включить |
| URL | `http://твой-сервер.com:7474/api/hikvision/event` |
| HTTP Method | POST |
| HTTP Authentication | None |

> **Порт**: используй HTTP порт сервера (7474 по умолчанию), не HTTPS (443) — устройства Hikvision часто не поддерживают проверку SSL сертификатов.

### Шаг 3 — Настроить события для отправки

**Путь**: Configuration → Event → Smart Event → Face Capture

Включить: Access Control Event → отправлять на HTTP server

### Шаг 4 — Настроить время

**Путь**: Configuration → System → System Settings → Time Settings

| Поле | Значение |
|------|---------|
| Time Zone | (GMT+05:00) Dushanbe / Tashkent |
| NTP Server | `pool.ntp.org` |
| Enable NTP | ✅ |

Время должно совпадать с сервером — иначе события будут с неверным временем.

### Шаг 5 — Статический IP (альтернатива DHCP Reservation)

Если хочешь задать IP прямо на устройстве:

**Путь**: Configuration → Network → Basic Settings → TCP/IP

| Поле | Значение |
|------|---------|
| DHCP | Выключить |
| IPv4 Address | `192.168.1.101` |
| Subnet Mask | `255.255.255.0` |
| Default Gateway | `192.168.1.1` |
| DNS Server | `8.8.8.8` |

## 10. Вкладка "Доступ к дверям" в профиле сотрудника

В профиле каждого сотрудника есть вкладка **"Доступ"** (СКУД).

### Что показывает:

- Список всех дверей компании
- Для каждой двери: **есть ли у этого сотрудника доступ** (реальная проверка в БД)
- Дата выдачи доступа и кто выдал
- Кнопки выдачи и отзыва

### Как работает кнопка "Выдать доступ":

```
Клик → POST /api/doors/:doorId/grant/:employeeId
    ↓
Сервер: создаёт DoorAccess (запись в БД)
Сервер: создаёт DoorCommand { action: "grant", status: "pending" }
    ↓
Relay-agent в офисе получает команду (через 5 секунд)
    ↓
Загружает сотрудника на Hikvision (пользователь + фото лица)
    ↓
Статус → "done" или "failed"
```

### Как работает кнопка "Отозвать":

```
Клик → DELETE /api/doors/:doorId/revoke/:employeeId
    ↓
Сервер: удаляет DoorAccess
Сервер: создаёт DoorCommand { action: "revoke", status: "pending" }
    ↓
Relay-agent: удаляет пользователя с Hikvision устройства
```

### Почему доступ нужно выдавать явно:

Системе нужно загрузить фото лица конкретного сотрудника на конкретное устройство.
Просто добавить сотрудника в систему недостаточно — Face ID знает только тех, кто явно синхронизирован.

### Синхронизация всех сотрудников на дверь:

На странице `/doors` у каждой двери есть кнопка **"Синхронизировать всех"** — создаёт команды для всех сотрудников компании у которых уже есть доступ к этой двери.

## 11. Чеклист для каждого офиса

### При добавлении нового офиса:

```
[ ] 1. Назначить статические IP устройствам (через роутер или на устройстве)
[ ] 2. Прописать устройства в HIKVISION_DEVICES на сервере (.env)
[ ] 3. Перезапустить backend: docker-compose up -d backend
[ ] 4. Создать офис в системе (Настройки → Офисы) — название должно совпадать с officeName в .env
[ ] 5. Добавить дверь в системе (/doors → Добавить дверь)
[ ] 6. На каждом устройстве: настроить HTTP Notification на адрес сервера
[ ] 7. На каждом устройстве: настроить временну́ю зону и NTP
[ ] 8. Проверить: подойти к устройству → лицо должно сработать → прийти Telegram
[ ] 9. [Опционально] Установить relay-agent для управления Face ID
[ ] 10. [Опционально] Выдать доступ сотрудникам через профиль или /doors
```

### Проверка после настройки:

1. **Тест Telegram**: в системе есть кнопка "Тест уведомление" (или подойти к устройству)
2. **Тест посещаемости**: зайти/выйти → посмотреть в разделе "Посещаемость" за сегодня
3. **Тест Face ID**: выдать доступ → дождаться `done` в очереди → проверить на устройстве через SADP

## 12. Частые проблемы и решения

### ❌ Telegram не приходит когда заходит сотрудник

**Проверь по шагам:**

1. IP устройства есть в `HIKVISION_DEVICES`?
   - `backend/.env` → `HIKVISION_DEVICES` → найди IP устройства

2. Backend перезапустили после изменения .env?
   ```bash
   docker-compose up -d backend
   ```

3. Устройство достигает сервера?
   - На устройстве: System → Network → проверь Default Gateway и DNS
   - Попробуй пинговать сервер из той же сети

4. Правильный URL в HTTP Notification?
   - Должно быть: `http://сервер:7474/api/hikvision/event`
   - Порт 7474, не 7373 (это frontend) и не 7272 (прямой backend без nginx)

5. skudId сотрудника заполнен?
   - В профиле сотрудника → поле "СКУД ID" должно совпадать с employeeNo на устройстве

---

### ❌ Посещаемость пишется, но офис не определяется (officeId = null)

Название офиса в `officeName` не совпадает с названием в системе.

**Решение**: Настройки → Офисы → проверь точное написание названия → скопируй точь-в-точь в `HIKVISION_DEVICES`

---

### ❌ Команды висят в статусе "pending" — доступ не выдаётся

Relay-agent не запущен или не может достучаться до сервера.

**Проверки:**
```bat
:: Запущен ли агент?
tasklist | findstr node

:: Логи агента
type C:\hrms-agent\agent.log

:: Тест вручную
node C:\hrms-agent\relay-agent.js
```

---

### ❌ Команды в "failed" с ошибкой "Timeout 192.168.x.x"

Агент не может достучаться до Hikvision устройства.

**Проверки:**
1. Агент и устройства в одной локальной сети?
2. Правильный ли IP прописан при добавлении двери?
3. Можно ли пинговать: `ping 192.168.1.101` с компьютера агента

---

### ❌ "HTTP 401" от Hikvision в логах агента

Неверный логин или пароль устройства.

**Решение**: `/doors` → найди дверь → кнопка "Изменить" → проверь Логин и Пароль

---

### ❌ Команды в "processing" зависли (агент упал в процессе)

Агент взял команду, упал, и статус застрял в "processing".

**Решение**: Перезапустить агент — он автоматически при старте проверяет "processing" команды и сбрасывает их в "pending".

---

### ❌ Неизвестный сотрудник — лицо распознаётся но сотрудник не найден

Значит `skudId` в профиле сотрудника не совпадает с `employeeNo` в устройстве.

**Решение**:
1. В Telegram посмотри "СКУД №: 00042" (число из сообщения)
2. Открой профиль сотрудника → вкладка "Основные данные"
3. Поле "СКУД ID" → вписать это число

## Схема офисов (пример для 7 офисов)

```
HIKVISION_DEVICES=[
  {"ip":"192.168.1.101","officeName":"Офис Душанбе",   "direction":"IN"},
  {"ip":"192.168.1.102","officeName":"Офис Душанбе",   "direction":"OUT"},
  {"ip":"192.168.2.101","officeName":"Склад №1",        "direction":"IN"},
  {"ip":"192.168.2.102","officeName":"Склад №1",        "direction":"OUT"},
  {"ip":"192.168.3.101","officeName":"Офис Худжанд",   "direction":"IN"},
  {"ip":"192.168.3.102","officeName":"Офис Худжанд",   "direction":"OUT"},
  {"ip":"192.168.4.101","officeName":"Цех производства","direction":"IN"},
  {"ip":"192.168.4.102","officeName":"Цех производства","direction":"OUT"},
  {"ip":"192.168.5.101","officeName":"Офис Куляб",     "direction":"IN"},
  {"ip":"192.168.5.102","officeName":"Офис Куляб",     "direction":"OUT"},
  {"ip":"192.168.6.101","officeName":"Магазин",        "direction":"IN"},
  {"ip":"192.168.6.102","officeName":"Магазин",        "direction":"OUT"},
  {"ip":"192.168.7.101","officeName":"Главный офис",   "direction":"IN"},
  {"ip":"192.168.7.102","officeName":"Главный офис",   "direction":"OUT"}
]
```

> В каждом офисе IP-адреса берутся из той локальной сети офиса.
> Если офисы в разных локальных сетях, IP-пространства могут пересекаться (192.168.1.x в каждом) — **это проблема только если все в одной большой сети**. Если каждый офис — отдельная сеть за своим роутером/NAT, пересечение не страшно, так как сервер видит только IP который сам указан в устройстве при отправке события (`ipAddress` в теле запроса от Hikvision).
